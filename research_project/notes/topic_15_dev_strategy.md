# Topic 15: Стратегия разработки

## Вопрос
Сверху вниз (сервер → воркер → VM) или снизу вверх (VM → воркер → сервер)?

## Сравнение

| Подход | Плюсы | Минусы |
|---|---|---|
| Top-down | Продуманная архитектура | Долго до результата, нечем тестировать |
| **Bottom-up** | **Быстрая проверка концепции** | Возможен рефакторинг |

## Рекомендация: Bottom-up

### Обоснование
1. Основные риски — на уровне VM (Venus, RAM, VNC latency, virtiofs)
2. RPC (shai) уже готов — нет смысла полировать до проверки VM
3. Быстрый видимый прогресс мотивирует

### Дорожная карта

**Фаза 1: VM + CS2 (proof of concept)**
- QEMU + Venus на одном хосте
- CS2 384x288
- VNC управление
- virtiofs шаринг
- Измерение RAM/CPU

**Фаза 2: Воркер (локальное управление)**
- Rust daemon + virt crate
- XML генерация с fingerprint
- VNC контроллер
- Steam автологин
- Параллельное управление N VM

**Фаза 3: Сервер + RPC**
- shai message types
- Очередь аккаунтов
- Heartbeat / status
- Steam Guard relay
- Dashboard

**Фаза 4: Масштабирование**
- Multi-worker deployment
- Install script
- KSM/zswap tuning
- AI inference
- Поведенческая рандомизация
