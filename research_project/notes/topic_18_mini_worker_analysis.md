# Topic 18: Mini-Worker Architecture — Нужен ли сервис внутри VM?

## Формулировка вопроса
Нужен ли мини-воркер (маленький сервис внутри каждой VM), который будет общаться с большим воркером на хосте, или можно обойтись полностью внешним управлением?

## Три архитектурных подхода

### Подход A: Полностью внешнее управление (без мини-воркера)
**Инструменты:** VNC (framebuffer + input), QMP (screenshots, keyboard/mouse), QEMU Guest Agent (файлы, команды)

**Как работает:**
1. Хост-воркер пишет session файлы через guest-agent (`guest-file-write`)
2. Хост-воркер запускает Steam через guest-agent (`guest-exec`)
3. VNC читает framebuffer для распознавания UI состояния
4. QMP `input-send-event` отправляет клавиатуру/мышь
5. OpenCV/AI на хосте анализирует экран и принимает решения

**Плюсы:**
- VM полностью "тупая" — только базовая ОС + Steam + CS2
- Нет дополнительного ПО для поддержки внутри VM
- Все логика на хосте — проще обновлять и отлаживать
- Меньше attack surface для VAC (нет подозрительных процессов в VM)

**Минусы:**
- Guest Agent: только polling, нет events из VM → хост
- VNC framebuffer: высокий overhead при постоянном чтении
- Невозможно получить структурированные данные из Steam (статус, ошибки)
- Guest-exec не даёт realtime stdout — только после завершения процесса
- Сложно диагностировать проблемы внутри VM
- Задержка: VNC screenshot → анализ → отправка команды = 100-500ms

### Подход B: Полноценный мини-воркер внутри VM
**Архитектура:** Rust/Python сервис внутри VM, общающийся с хост-воркером по сети или через virtio-serial

**Как работает:**
1. Мини-воркер стартует при загрузке VM (systemd service)
2. Принимает команды от хост-воркера: "войди в аккаунт X", "запусти CS2"
3. Напрямую манипулирует файлами Steam (config.vdf, loginusers.vdf)
4. Мониторит процессы Steam/CS2, отправляет статус наверх
5. Может читать логи Steam и CS2 для структурированных ошибок

**Плюсы:**
- Прямой доступ к файловой системе Steam
- Может мониторить процессы (steam, cs2) в реальном времени
- Структурированная обратная связь (статус, ошибки, health checks)
- Быстрая реакция на события внутри VM
- Может генерировать TOTP коды для Steam Guard
- Может управлять Steam через его IPC механизм

**Минусы:**
- Дополнительный процесс в VM → потенциально детектируется VAC
- Нужно поддерживать и обновлять код внутри VM
- Усложняет процесс создания VM (нужно инжектить бинарник)
- Сетевая коммуникация между VM и хостом — дополнительная конфигурация
- Потенциально увеличивает потребление RAM

### Подход C: Гибридный (РЕКОМЕНДУЕМЫЙ)
**Архитектура:** QEMU Guest Agent + минимальные systemd-скрипты внутри VM

**Как работает:**
1. Хост использует guest-agent для записи session файлов
2. systemd service внутри VM запускает Steam при загрузке
3. Хост использует VNC для визуального управления (матч, геймплей)
4. Guest-agent для диагностики (проверка процессов, логов)
5. Простой bash/python скрипт как systemd unit для lifecycle Steam

**Плюсы:**
- Минимальный footprint внутри VM
- Guest-agent — стандартный компонент, не вызывает подозрений
- systemd service — стандартный способ автозапуска
- Не нужен сетевой канал связи (virtio-serial для guest-agent)
- Хост-воркер остаётся центром принятия решений
- Простота обновления скриптов через guest-agent

**Минусы:**
- Скрипты внутри VM всё же нужно поддерживать
- Нет realtime обратной связи (только polling через guest-agent)

## Анализ конкретных задач

### Задача 1: Инжекция Steam сессии
| Метод | Сложность | Надёжность | Скорость |
|-------|-----------|------------|----------|
| Guest-agent file-write | Низкая | Высокая | Мгновенно |
| Мини-воркер с RPC | Средняя | Высокая | Мгновенно |
| VNC автоматизация | Высокая | Низкая | 5-30 сек |
| Монтирование qcow2 на хосте | Низкая | Высокая | До загрузки VM |

**Вывод:** Guest-agent (`guest-file-write`) полностью решает задачу инжекции файлов. Мини-воркер не даёт преимуществ.

### Задача 2: Запуск Steam и CS2
| Метод | Сложность | Надёжность | Контроль |
|-------|-----------|------------|----------|
| Guest-agent guest-exec | Низкая | Средняя | Нет realtime |
| systemd autostart | Низкая | Высокая | Нет динамики |
| Мини-воркер | Средняя | Высокая | Полный |
| VNC автоматизация | Высокая | Низкая | Визуальный |

**Вывод:** systemd service (auto-login в Steam) + guest-exec для динамических задач. Мини-воркер даёт преимущество только в realtime мониторинге процесса.

### Задача 3: Переключение аккаунтов
| Метод | Сложность | Надёжность | Время |
|-------|-----------|------------|-------|
| Перезагрузка VM с новым overlay | Низкая | Высокая | 30-60 сек |
| Guest-agent: kill steam + write files + restart | Средняя | Высокая | 10-20 сек |
| Мини-воркер: graceful switch | Средняя | Высокая | 5-10 сек |
| VNC: визуальный logout + login | Очень высокая | Низкая | 30-60 сек |

**Вывод:** Guest-agent вполне достаточен. Мини-воркер экономит 5-10 секунд за счёт graceful управления процессами, но это не критично.

### Задача 4: Мониторинг состояния Steam/CS2
| Метод | Сложность | Реактивность | Детализация |
|-------|-----------|-------------|-------------|
| VNC (визуальный анализ) | Высокая | 100-500ms | Только экран |
| Guest-agent polling | Низкая | 1-5 сек | Процессы, логи |
| Мини-воркер | Средняя | <100ms | Полная |

**Вывод:** Мини-воркер даёт лучшую реактивность и детализацию. Но guest-agent polling с интервалом 2-5 сек достаточен для фарма.

### Задача 5: Steam Guard (TOTP)
| Метод | Сложность | Надёжность |
|-------|-----------|------------|
| Генерация на хосте + ввод через VNC | Средняя | Средняя |
| Генерация на хосте + guest-exec | Низкая | Высокая |
| Мини-воркер с steam-totp | Средняя | Высокая |
| LoginApprover (программное подтверждение) | Средняя | Высокая |

**Вывод:** LoginApprover или генерация TOTP на хосте + guest-exec. Мини-воркер не нужен.
