# Topic 14: Implementation Order — Top-Down vs Bottom-Up

## Контекст

Система состоит из трёх уровней:
1. **Сервер** (license repo — Axum + SeaORM + Teloxide) — управление лицензиями, очередь аккаунтов
2. **Воркер** (новый Rust-сервис на хосте) — управление KVM/QEMU, создание/уничтожение VM
3. **VM** (внутри гостевой ОС) — Steam, CS2, мини-агент для приёма команд

## Сравнительная таблица

| Критерий | Top-Down (Сервер → Воркер → VM) | Bottom-Up (VM → Воркер → Сервер) |
|---|---|---|
| **Первый результат** | Позже (нужно написать сервер, протокол, воркер, прежде чем увидеть CS2) | **Быстрее** (CS2 в VM работает через дни) |
| **Проверка концепции** | Откладывается до интеграции всех слоёв | **Сразу** (можно проверить Venus GPU, zswap, Steam login) |
| **Риск "мёртвого кода"** | Высокий — можно написать API для сценариев, которые не работают в VM | Низкий — каждый слой проверен перед следующим |
| **Архитектурная чистота** | Чище — API проектируется до реализации | Может потребоваться рефакторинг API под реальные ограничения VM |
| **Мотивация команды** | Низкая на начальном этапе (работа без видимого результата) | **Высокая** (видимый прогресс сразу) |
| **Интеграционные проблемы** | Обнаруживаются поздно | **Обнаруживаются рано** |
| **Параллельная работа** | Возможна (разные люди на разных слоях) | Сложнее (каждый слой зависит от предыдущего) |
| **Зависимость от железа** | Не нужно железо для начала работы | Нужен хост с GPU и KVM для проверки VM |

## Вывод: Bottom-Up (VM → Воркер → Сервер)

**Рекомендуется начинать снизу** по следующим причинам:

1. **Критические технические риски находятся внизу.** Venus GPU в QEMU 9.2+, работа CS2 при 500 MB (или реальный минимум), zswap + KSM в связке, автоматический вход в Steam — всё это гипотезы, которые нужно проверить до написания сервера. Если Venus не даёт стабильного Vulkan для CS2, вся архитектура меняется.

2. **Сервер уже существует частично.** License repo (Axum + SQLite) уже имеет heartbeat, сессии, скачивание билдов. Его можно расширить позже, когда точно понятно, какие команды нужны воркеру.

3. **Воркер — самый неизученный компонент.** Управление VM через `virt` crate, VNC-захват экрана, ввод команд — это новый код, и его нужно прототипировать и тестировать раньше всего.

## Рекомендуемый порядок реализации

### Фаза 1: Proof of Concept в VM (1–2 недели)
1. Установить QEMU 9.2+ с Venus на хост
2. Создать базовый образ VM (Debian 12 minimal + OpenBox)
3. Проверить работу Venus GPU (Vulkan) в гостевой ОС
4. Установить Steam + CS2 внутри VM
5. Проверить запуск CS2 в 384×288 с Vulkan
6. Измерить реальное потребление RAM (CS2 + ОС)
7. Настроить zswap + KSM на хосте, запустить 2–3 VM параллельно
8. Протестировать автоматический Steam login через `steamguard` crate

### Фаза 2: Воркер (Rust-сервис на хосте) (2–3 недели)
1. Создать Rust-проект воркера с `virt` crate для управления VM
2. Реализовать создание/уничтожение VM из шаблона XML
3. Реализовать VNC-захват экрана и отправку ввода
4. Создать конвейер: запуск VM → ожидание загрузки → логин в Steam → запуск CS2
5. Реализовать мониторинг состояния VM (heartbeat, crashdetection)
6. Добавить параллельное управление N VM

### Фаза 3: Интеграция с сервером (1–2 недели)
1. Расширить license repo: добавить API для воркеров (регистрация, получение аккаунтов, отчёт статуса)
2. Реализовать протокол воркер ↔ сервер (REST или WebSocket)
3. Добавить очередь аккаунтов (Redis или встроенная в SQLite)
4. Интеграционное тестирование: сервер → воркер → VM → CS2

### Фаза 4: Продакшн (1–2 недели)
1. Автоматизация деплоя воркера (install script)
2. Мониторинг (Prometheus + Grafana)
3. Обработка ошибок, переподключение, авто-восстановление
4. Масштабирование на несколько хостов
